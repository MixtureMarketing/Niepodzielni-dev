A. PRIORYTET HIGH — natychmiast (zysk największy / małe ryzyko)
A1. Brak composite indexu wp_postmeta(post_id, meta_key)
Lokalizacja: database-template.sql:567-569 — tylko KEY post_id i KEY meta_key(191).
Problem: Każde WP_Query z meta_query (psychologowie: specjalizacje, nurty, języki, Bookero ID) i Carbon Fields wymusza pełny skan postmeta lub merge join.
Co zrobimy w PR (bez dotykania prod):
Test na dev (Docker): wp db query "ALTER TABLE wp_postmeta ADD INDEX idx_post_meta (post_id, meta_key(191));"
Pomiar EXPLAIN przed/po (3 typowe meta_query) i czas odpowiedzi listingu psychologów.
Dodać do database-template.sql:567-569 index w schemacie (nowe instalacje od razu mają go).
Stworzyć migration runner: web/app/mu-plugins/niepodzielni-core/migrations/2026-05-add-postmeta-index.php z idempotentnym if (!index_exists()) ALTER. Uruchamiany przez WP-CLI: wp np migrate run. Operator wykona ręcznie na prod w oknie serwisowym.
Weryfikacja (dev): EXPLAIN SELECT * FROM wp_postmeta WHERE post_id = X AND meta_key = 'specjalizacje' → Using index.
Wpływ: −500 do −2000 ms na listingach z meta_query.

Pliki:

web/app/mu-plugins/niepodzielni-core/api/40-opinie-api.php:283

Naprawa: Cache filemtime w wp_cache (kluczem ścieżka) lub użyć stałej wersji z manifestu Vite/composer:

if ($ver === false) {
