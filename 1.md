To kluczowe pytanie architektoniczne — Zaraz i GTM się dublują (oba są tag managerami), więc masz trzy realne ścieżki:

1. Zaraz zastępuje GTM (najczystsze)
Przenosisz tagi (GA4, Meta Pixel, Ads) z GTM do Zaraz. GTM znika ze strony. Zaraz Consent Manager natywnie wysyła sygnały Consent Mode v2 do GA4/Ads server-side. Najszybsze, najmniej skryptów na froncie, ale wymaga przekonfigurowania tagów (zwykle 1-2 dni roboty dla typowego site'u).

2. GTM zostaje, ładowany jako jeden tag w Zaraz
Zaraz odpala gtm.js dopiero po zgodzie na analytics_storage. Problem: Zaraz CM nie kontroluje co GTM robi w środku — tagi w GTM nadal potrzebują własnej obsługi Consent Mode v2 (przez gtag('consent', ...) w GTM). Czyli i tak musisz ogarnąć consent w GTM, a Zaraz CM pełni rolę „bramki on/off" dla całego GTM. Działa, ale dwa systemy consent.

3. GTM zostaje, Consent Mode v2 robisz w GTM (klasycznie)
Wtedy Zaraz Consent Manager jest zbędny — bierzesz CookieYes/Cookiebot/Klaro, który wstrzykuje gtag('consent', 'default'/'update', ...) przed GTM. To ścieżka, którą miałeś.

Rekomendacja: jeśli marketing nie zarządza aktywnie tagami w GTM (czyli Ty/dev to robicie), idź w opcję 1 — Zaraz zastępuje GTM w całości. Jeśli marketing siedzi w GTM i nie chcesz im tego zabierać, opcja 3 z normalnym CMP jest mniej bolesna niż opcja 2.

Tak, słusznie pytasz — to jest często mylone, są trzy różne poziomy:

Trzy poziomy "server-side"
1. Browser → Zaraz (to co macie teraz)
Event startuje w przeglądarce (zaraz.track()), Zaraz na edge'u Cloudflare wysyła do GA4/Meta. Lepsze niż GTM client-side (jeden request, nie 10), ale jeśli ad-blocker zablokuje zaraz.js lub user zamknie przeglądarkę za szybko — event przepada.

2. Zaraz + Meta CAPI (hybryda przez Zaraz)
W Zaraz dashboardzie dodajesz Meta Conversions API jako drugie narzędzie obok Meta Pixel. Ten sam zaraz.track('purchase', ...) leci równolegle do pixel (browser) i CAPI (server). Meta deduplikuje po event_id. Łatwe w konfiguracji, ale nadal zależne od tego że user dotrze do thank-you page.

3. Pure server-to-server (S2S) z PHP — to o czym słyszałeś
Event wysyłany wyłącznie z backendu, nigdy nie dotyka przeglądarki. Dla Was kluczowe miejsca:

Bookero purchase — webhook od bramki płatniczej (tpay/p24) → WP action → PHP wysyła do GA4 Measurement Protocol + Meta CAPI
Donation — to samo, webhook od bramki
Lead form — po zapisie do DB
Zalety S2S:

ad-blocker nie zablokuje (request leci z PHP, nie z przeglądarki)
iOS Safari / ITP / brak third-party cookies nie wpływa
user może zamknąć kartę po zapłacie — event i tak poleci
realna odzysk 20–40% konwersji które normalnie giną
Rekomendacja dla Was
Hybryda: Zaraz dla wszystkich eventów + S2S dla purchase i donation.

purchase event:
  ├── Browser:  zaraz.track('purchase', { event_id: 'BKR-1234', ... })
  └── Server:   PHP hook na 'bookero_booking_complete'
                  ├── GA4 Measurement Protocol (POST mp/collect)
                  └── Meta CAPI (POST /events)
                Oba z tym samym event_id → automatyczna deduplikacja
Macie już transaction_id = 'BKR-' + Date.now() — to idealny event_id do deduplikacji, wystarczy wymóc ten sam ID po obu stronach.

Context
Niepodzielni przechodzą z architektury GTM/CookieYes (wtyczki WP) na Cloudflare Zaraz jako jedyny tag manager. Stan obecny:

Zaraz tracking jest częściowo wdrożony — eventy Bookero (bookero-init.js:160-249) i AI chat (ai-chat.js:811) lecą przez window.zaraz.track() z fallbackiem do dataLayer.

Większość konwersji nie jest śledzona: matchmaker, formularz kontaktowy, listing/profil psychologa, event reminders — zero eventów. Donations w fazie scaffolding (faza C niedokończona).
