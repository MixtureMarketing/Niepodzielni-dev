# Plan wdrożenia monitoringu — co robię sam, co zostawiam

## Context

Po PR #11 (`docs/monitoring-runbook.md` na branchu `claude/monitoring-runbook`)
mamy ułożony plan obserwowalności. Teraz wdrożenie. User pyta:

1. **Czy mogę zrobić to sam, bez dostępu do produkcyjnego VPS?**
2. **Co konkretnie muszę zostawić Claude'owi z dostępem do serwera /
   panelu admina?**

Odpowiedź: tak, dużą część kodu mogę zrobić sam — przygotuję 3 PR-y na
gałąź `dev` (PR #7 security został zmergowany właśnie do `dev`, więc to
jest właściwa baza). Reszta — instalacje na VPS, sekrety, panele SaaS —
wymaga dostępu, którego nie mam, i opisuję to jako TODO w PR description
+ checklist dla wykonawcy.

### Czego nie mam dostępu

- **Produkcyjny VPS** `185.201.113.217` (`new.niepodzielni.com`) — SSH/sudo.
- **Trellis vault** — hasło `ANSIBLE_VAULT_PASS` (potrzebne do edycji
  zaszyfrowanych `group_vars/{production,staging}/vault.yml`).
- **Konta SaaS**: Discord, Sentry, Better Stack, Netdata Cloud,
  Cloudflare dashboard.
- **`wrangler secret put`** dla Workera (wymaga zalogowanego CLI Cloudflare).

### Co mam

- Pełny dostęp do repo `MixtureMarketing/Niepodzielni-dev`.
- Mogę tworzyć branche, commitować, pushować, otwierać PR-y, edytować
  ich opisy.
- Lokalnie: PHP 8.4, Composer, Node, npm, Pint, PHPStan, Pest — czyli
  pełna walidacja każdego commita.

### Strategia

Dzielę pracę na **3 małe PR-y** zamiast jednego dużego — każdy:

- jest izolowany do jednej warstwy,
- ma osobny test plan i checklist dla wykonawcy,
- może być merged niezależnie, w dowolnej kolejności (są ortogonalne).

Wszystkie 3 PR-y idą **bazą na `dev`**, nie na `main`, bo aktualny stan
mu-pluginów (audit log, login throttle) jest na `dev` po merge PR #7.

## PR-A — Audit digest + brute-force burst alert

### Po co

Codzienny digest bezpieczeństwa do Discorda + email; natychmiastowy alert
gdy >20 lockoutów loginu w 1h. Bazuje na istniejącej tabeli `wp_np_audit`
(z `14-audit-log.php`, PR #7) i hooku `np_security_lockout` (z
`13-login-throttle.php`, PR #7). Najbezpieczniejszy PR w pakiecie — sam
PHP, zero zewnętrznych zależności poza HTTP do Discord webhook.

### Branch

`claude/monitoring-audit-digest` na bazie `dev`.

### Co tworzę (~120 linii PHP, 3 pliki, 2 modyfikacje)

1. **`web/app/mu-plugins/niepodzielni-core/admin/17-audit-digest.php`** (NEW, ~80 linii)
   - WP-Cron `np_audit_digest_daily` codziennie o 7:00.
   - Czyta z `wp_np_audit` zliczone events z 24h: `login_success`,
     `login_failed`, `login_lockout`, `password_reset`, `user_registered`.
   - Top 5 IP po `login_failed`.
   - Format raportu jako tekst monospace (Discord ``` block).
   - `wp_mail()` do `admin_email` (backup).
   - `wp_remote_post()` do `NP_DISCORD_WEBHOOK_URL` z payloadem
     `{ username: "WordPress Audit", content: "```\n...\n```" }`.
   - `register_deactivation_hook` — bezpieczne wyrejestrowanie crona.

2. **`web/app/mu-plugins/niepodzielni-core/admin/19-lockout-burst-alert.php`** (NEW, ~30 linii)
   - Hook `np_security_lockout` (już emitowany przez `13-login-throttle.php`).
   - Sliding window transient `np_lockout_burst` (TTL 1h).
   - Po przekroczeniu 20 → jednorazowy alert do Discord + flag w
     transient żeby nie spamować dopóki okno nie wygaśnie.

3. **`web/app/mu-plugins/niepodzielni-core.php`** (modify)
   - `require_once` dla `17-audit-digest.php` i `19-lockout-burst-alert.php`
     po `15-retention-cron.php`.

4. **`config/application.php`** (modify)
   - `Config::define('NP_DISCORD_WEBHOOK_URL', env('NP_DISCORD_WEBHOOK_URL') ?: '');`
     (po linii z `NP_AI_BOT_TOKEN`).

5. **`.env.example`** (modify)
   - Dopisać `NP_DISCORD_WEBHOOK_URL=` + komentarz z linkiem do runbooka.

### Walidacja przed PR

- `php -l` na obu nowych plikach.
- `vendor/bin/pint --test`.
- `vendor/bin/phpstan analyse`.
- `vendor/bin/pest` — bez nowych testów (cron sam siebie waliduje przez
  `wp cron event run`).

### Co zostawiam wykonawcy (Claude z dostępem do VPS / admin)

W PR description jako **„Operator checklist"**:

1. **Discord** (Admin, 5 min):
   - Wygeneruj webhook „WordPress Audit" w kanale `#alerts-niepodzielni`
     (instrukcje: `docs/monitoring-runbook.md` sekcja 2).
   - Skopiuj URL.

2. **Trellis vault — production** (wymaga dostępu do hasła vault):
   ```bash
   cd trellis
   ansible-vault edit group_vars/production/vault.yml
   ```
   Dopisać do `vault_wordpress_sites['new.niepodzielni.com']['env']`:
   ```yaml
   np_discord_webhook_url: 'https://discord.com/api/webhooks/.../...'
   ```

3. **Trellis vault — staging** (analogicznie, jeśli chcemy alerty z dev):
   ```bash
   ansible-vault edit group_vars/staging/vault.yml
   ```
   To samo pole. Można użyć **innego webhooka** żeby alerty z dev nie
   mieszały się z prod (np. „WordPress Audit DEV" w osobnym kanale
   `#alerts-niepodzielni-dev`) — albo użyć tego samego webhooka.

4. **Trellis `wordpress_sites.yml` — przekazanie env do PHP** (modify
   w PR przeze mnie):
   ```yaml
   env:
     ...
     np_discord_webhook_url: "{{ vault_wordpress_sites['new.niepodzielni.com']['env']['np_discord_webhook_url'] }}"
   ```
   Analogicznie w staging.

   > **Uwaga**: ta zmiana w `wordpress_sites.yml` jest **w PR**, ale
   > efekt zadziała dopiero po deployu Ansible.

5. **Deploy** (wykonawca z dostępem):
   ```bash
   cd trellis
   ansible-playbook deploy.yml -e env=production -e site=new.niepodzielni.com
   ```

6. **Smoke test** (wykonawca, ~2 min):
   ```bash
   ssh root@new.niepodzielni.com
   cd /srv/www/new.niepodzielni.com/current/web/wp
   wp cron event run np_audit_digest_daily
   ```
   Discord powinien dostać wiadomość od „WordPress Audit". Jeśli nie —
   sprawdź `wp option get np_discord_webhook_url` (powinno być URL z
   vault) i error log nginx/PHP.

### Ryzyka i rollback

- **Brak ryzyka dla użytkowników** — to czyste read-only z DB +
  fire-and-forget HTTP POST. Failure → silent (catch i kontynuuj).
- **Rollback**: `git revert` PR-a, `ansible-playbook deploy.yml` ponownie.
- **Spam Discord** przy pierwszym uruchomieniu — pierwszy digest pokaże
  zdarzenia z ostatnich 24h, więc duży raport jednokrotnie. Każdy
  kolejny dzień normalny.

## PR-B — Sentry: PHP + Worker

### Po co

Centralny zbieracz błędów z aplikacji WP i Workera. Sentry grupuje
identyczne błędy, pokazuje stack trace, mapuje na user / release /
environment. PR jest większy bo dotyka 2 stacków (PHP + TypeScript) i
wprowadza zewnętrzną zależność (`sentry/sentry`, `@sentry/cloudflare`).

### Branch

`claude/monitoring-sentry` na bazie `dev`.

### Co tworzę

**Strona PHP**:

1. **`composer.json`** (modify):
   - Dodać `"sentry/sentry": "^4.0"` w `require`.
   - `composer update sentry/sentry --lock` żeby zaktualizować
     `composer.lock` (zaciągam tylko ten pakiet i jego deps).

2. **`web/app/mu-plugins/niepodzielni-core/admin/18-sentry-init.php`** (NEW, ~50 linii):
   - Early return jeśli `SENTRY_DSN` puste (dev defaults).
   - `Sentry\init()` z `environment` z `WP_ENV`, `release` z `NP_RELEASE`.
   - `traces_sample_rate: 0.1` (10% dla performance).
   - **`before_send` filter**: wyrzuca `request.data` (POST body) i
     `cookies` z eventu — żeby PII z formularza kontaktowego nie wyciekło.

3. **`web/app/mu-plugins/niepodzielni-core.php`** (modify):
   - `require_once` dla `18-sentry-init.php` po `13-login-throttle.php`.

4. **`config/application.php`** (modify):
   - `Config::define('SENTRY_DSN', env('SENTRY_DSN') ?: '');`
   - `Config::define('NP_RELEASE', env('NP_RELEASE') ?: 'dev');`

5. **`.env.example`** (modify):
   - `SENTRY_DSN=` + komentarz z odsyłaczem do runbooka (sekcja 4.3).
   - `NP_RELEASE=`

**Strona Worker**:

6. **`workers/ai-agent/package.json`** (modify):
   - Dodać `"@sentry/cloudflare": "^9"` w `dependencies`.
   - `npm install --package-lock-only` żeby zaktualizować
     `package-lock.json` bez instalacji w `node_modules` (stay clean).

7. **`workers/ai-agent/src/index.ts`** (modify):
   - Import `import * as Sentry from '@sentry/cloudflare';`.
   - Owinąć handler: `export default Sentry.withSentry(env => ({ dsn: env.SENTRY_DSN, ...}), handler)`.
   - Albo wzorzec z opcją (jeśli `withSentry` API się zmieniło) — użyję
     `Sentry.init` w pierwszym requeście jeśli `env.SENTRY_DSN`. Wybór
     finalny po sprawdzeniu wersji 9 SDK.

8. **`workers/ai-agent/src/types.ts`** (modify):
   - Dodać `SENTRY_DSN: string;` i `SENTRY_ENV: string;` w `interface Env`.

9. **`workers/ai-agent/wrangler.toml`** (modify):
   - W komentarzu nad `[vars]`: dopisać że `SENTRY_DSN` i `SENTRY_ENV`
     muszą być ustawione przez `wrangler secret put`.

### Walidacja przed PR

- `composer install` lokalnie + `vendor/bin/phpstan analyse` —
  Sentry ma typehinty, PHPStan musi przejść.
- `vendor/bin/pint --test`.
- `cd workers/ai-agent && npm install && npx tsc --noEmit` — exit 0.
- `vendor/bin/pest` — testy nadal przechodzą.

### Co zostawiam wykonawcy

1. **Sentry — utworzenie projektów** (~10 min):
   - https://sentry.io → Create Project → PHP (Generic, nie
     WordPress) → nazwa `niepodzielni-wp`.
   - Skopiuj DSN.
   - Drugi projekt: Cloudflare Workers → `niepodzielni-worker`.
     Skopiuj DSN.
   - **Ważne**: dla dev/prod osobne projekty (a nie environments
     w jednym projekcie) — Sentry free tier daje pełne 5k errors/mies
     per projekt, więc lepiej rozdzielać niż dzielić quotę.

2. **Trellis vault — production** + **staging**:
   ```bash
   ansible-vault edit group_vars/production/vault.yml
   # dopisać:
   #   sentry_dsn: 'https://abc@sentry.io/123'   (prod DSN)
   #   np_release: '1.0.0'                        (lub commit SHA z deploy)
   ```

3. **`wordpress_sites.yml`** — przekazanie env (przeze mnie w PR):
   ```yaml
   env:
     ...
     sentry_dsn: "{{ vault_wordpress_sites['new.niepodzielni.com']['env']['sentry_dsn'] }}"
     np_release: "{{ vault_wordpress_sites['new.niepodzielni.com']['env']['np_release'] }}"
   ```

4. **Worker secrets** (wykonawca z `wrangler` zalogowanym do CF):
   ```bash
   cd workers/ai-agent
   wrangler secret put SENTRY_DSN --env production
   # wkleić DSN z projektu niepodzielni-worker
   wrangler secret put SENTRY_ENV --env production
   # wpisać: production
   wrangler secret put SENTRY_DSN --env staging
   wrangler secret put SENTRY_ENV --env staging
   # wpisać: staging
   ```

5. **Sentry → Discord webhook**:
   - Sentry → Settings → Integrations → **Webhooks (Legacy)** → Add.
   - URL: `https://discord.com/api/webhooks/.../slack` (webhook
     **„Sentry"** z runbooka sekcja 2 + sufiks `/slack` żeby Discord
     zrozumiał format Slacka).
   - Project → Alerts → Create Alert Rule:
     - Trigger: „A new issue is created" (tylko nowe, nie powtórki).
     - Action: „Send a notification via Webhook" → wybierz właśnie
       dodany.

6. **Deploy**:
   - WP: `ansible-playbook deploy.yml -e env=production`
   - Worker: `cd workers/ai-agent && wrangler deploy --env production`

7. **Smoke test PII**:
   - Po deploy wpisz świadomie błędne dane do `/wp-json/niepodzielni/v1/forms/contact/submit`
     (np. brakujący wymagany field) → wyzwoli walidację po stronie WP →
     event w Sentry. Sprawdź **„Additional data"** — body POST powinien
     być `[Filtered]` lub w ogóle nieobecny. Jeśli widać `email`/`phone`
     w stack trace — `before_send` nie zadziałał, fix przed produkcją.

### Ryzyka i rollback

- **Performance**: `traces_sample_rate: 0.1` znaczy 10% requestów
  zbiera trace — minimalny narzut. Można zmienić na 0.01 (1%) jeśli
  zauważymy spowolnienie.
- **PII leak**: jedyne realne ryzyko. Smoke test (krok 7) musi być przed
  rolloutem.
- **Sentry quota**: 5k errors/mies. Przy serii błędów po deployu można
  szybko wypalić. Sentry ma „Inbound filters" — sugeruję wyłączyć
  „Health Check Failures" i bot user-agents.
- **Rollback**: jak PR-A. Plus odpiąć `Sentry.withSentry` w Worker
  i `wrangler deploy --env production` od nowa.

### Po co

Plugin do debug wydajności (per-request lista zapytań SQL, czas, hooki).
Tylko na dev. Plus twardy bezpiecznik na produkcji żeby nie pokazał się
nigdy.

### Branch

`claude/monitoring-query-monitor` na bazie `dev`.

### Co tworzę (~10 linii)

1. **`composer.json`** (modify):
   - Sprawdzić, czy są `"repositories": [{"type": "composer", "url": "https://wpackagist.org"}]`.
     Jeśli nie — dodać.
   - W `require-dev` dopisać `"wpackagist-plugin/query-monitor": "^3"`.

2. **`composer.lock`** (modify):
   - `composer update wpackagist-plugin/query-monitor --lock`.

3. **`config/environments/production.php`** (modify):
   - Dopisać na końcu:
     ```php
     // Bezpieczeństwo: jeśli Query Monitor by się znalazł na produkcji
     // (np. przez auto-update), nie pokazuj jego output nikomu.
     if (! defined('QM_DISABLED')) {
         define('QM_DISABLED', true);
     }
     ```

### Walidacja

- `composer install` lokalnie.
- `composer audit` — Query Monitor stabilny.
- `vendor/bin/phpstan analyse` — bez nowych błędów.

### Co zostawiam wykonawcy

1. **Deploy na dev**:
   ```bash
   cd trellis
   ansible-playbook deploy.yml -e env=staging -e site=dev.niepodzielni.com
   ```
   Po deploy: Trellis robi `composer install --no-dev` na produkcji
   i `composer install` (z dev deps) na staging. Query Monitor wleci
   tylko na dev.

2. **Aktywacja** (jeśli nie auto-aktywuje się):
   ```bash
   ssh deploy@dev.niepodzielni.com
   cd /srv/www/dev.niepodzielni.com/current/web/wp
   wp plugin activate query-monitor
   ```

3. **Smoke test**:
   - Zaloguj się do `https://dev.niepodzielni.com/wp-admin` jako admin.
   - Otwórz dowolną podstronę (np. listing psychologów).
   - W pasku admina pojawia się panel QM z liczbą zapytań SQL i czasem.

4. **Smoke test produkcji** (sanity check że nie wyciekło):
   - `https://new.niepodzielni.com` → pasek admina (jako admin)
     **nie powinien** mieć panelu QM. Jeśli ma — rollback i debug
     `composer install --no-dev` deploy hookow.

### Ryzyka

- Bardzo małe — plugin tylko dla zalogowanych adminów, dev-only via
  composer flag, plus `QM_DISABLED=true` na prod.
